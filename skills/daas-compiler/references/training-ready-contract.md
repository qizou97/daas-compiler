# Training-Ready Contract

## Definition

A dataset is **training-ready** (L4) only when it can be consumed directly by the
intended training loader **without** any of the following additional steps:

- Preprocessing (normalization, filtering, gene reordering)
- Joining (merging manifest with h5ad)
- Splitting (train/val/test materialization)
- Gene reordering (aligning expression vectors to a fixed gene panel)
- Image conversion (JPEG → tensor)
- Target conversion (raw counts → normalized expression, sparse → dense)
- Artifact conversion (h5ad → tensor, parquet → index)

If the training code must perform any of these steps, the dataset is not training-ready.

## Core Invariants

These invariants must hold for any L4 training-ready dataset:

- Gene order is fixed and referenced by `gene_panel.json`.
- `gene_panel.sha256` is written and cross-referenced in per-cell metadata.
- For L3 compiled artifacts: manifest row `i` maps to expression row `i` (`global_idx` stable).
- WDS per-cell metadata references `global_idx`, `sample_id`, `cell_id`, and `gene_panel_sha256`.
- **Splits are a metadata layer.** Split membership is determined by `splits/split_membership.parquet`
  and the per-split JSON lists (`splits/train.json`, etc.). Shards are not physically partitioned
  by split by default.
- The loader filters by split metadata at runtime without requiring physical shard duplication.

## Split Policy

> **Splits are metadata, not physical shard partitions.**

The default training-ready layout stores all cells in a single canonical set of shards
under `data/`. The loader selects which cells belong to train, val, or test at runtime by
consulting `splits/split_membership.parquet`.

**Physical train/val/test shard directories are an optional export mode only.**
They must not be produced by default, and must require an explicit flag
(e.g., `--materialize-split-shards`) to trigger. Tools and documentation must not describe
physically-partitioned shard directories as the default or required L4 layout.

### Generated split policies

DAAS generates splits at the **sample level or group level only**. Supported policies:

| Policy | Assignment unit | Leakage protection |
|---|---|---|
| `sample_holdout` | `sample_id` | All cells from same sample stay in one split |
| `ratio_by_group` | any group column (`sample_id`, `patient_id`, `donor_id`, `slide_id`, `batch_id`) | All cells from same group stay in one split |
| `group_kfold` | any group column | Each group belongs to exactly one fold |
| `existing_file` | external file (may be `global_idx`-level) | DAAS emits leakage warning for `global_idx`-level files |

**DAAS does not generate random cell-level train/val/test splits.**
`split_membership.parquet` may be indexed by `global_idx` for loader efficiency,
but every row generated by DAAS inherits its split assignment from the sample/group
assignment. Individual cells are never randomly scattered across splits by DAAS.

If a user requests a random cell-level split, the agent must:
1. Explain that DAAS does not generate cell-level random splits because they can
   cause sample/patient leakage.
2. Suggest `sample_holdout` or `ratio_by_group` instead.
3. Offer `existing_file` only if the user provides their own benchmark split.

## Explicit Non-Examples

| Artifact | Level | Why NOT training-ready |
|---|---|---|
| Per-sample JPEG shards + expression.h5ad + manifest.parquet | L2 | Requires joining, splitting, global gene panel |
| Compiled global manifest.parquet + expression.h5ad | L3 | Requires splitting, task config, loader config |
| Bundled WDS shards without splits/ or gene_panel.json | L3b | Incomplete: missing split metadata and task metadata |
| Physically partitioned train/ val/ test/ shard directories (no split_membership) | L4-export | Optional export variant, not the canonical default layout |

## HE2ST Training-Ready Requirements

A dataset compiled for H&E → Spatial Transcriptomics prediction is training-ready when
ALL of the following artifacts are present and consistent:

### Default canonical layout

```
{task_output}/
  data/
    shard-000000.tar
    shard-000001.tar
    ...
    bundled_manifest.parquet   (optional: per-shard cell index)

  splits/
    train.json                 ordered list of global_idx values → train
    val.json                   ordered list of global_idx values → val
    test.json                  ordered list of global_idx values → test
    split_membership.parquet   per-cell: global_idx, sample_id, cell_id, split
    split_report.json          method, seed, per-sample per-split counts

  gene_panel.json
  gene_panel.sha256
  task_config.yaml
  loader_config.yaml
  dataset_card.json
  validation_report.json
```

### Alternative reuse layout (compiled/ as base)

When reusing an existing `compiled/` directory:

```
compiled/
  manifest.parquet
  expression.h5ad
  gene_panel.json
  gene_panel.sha256
  {sample_id}/
    shard-*.tar

  splits/
    he2st_train.json
    he2st_val.json
    he2st_test.json
    he2st_split_membership.parquet
    he2st_split_report.json

  tasks/
    he2st/
      task_config.yaml
      loader_config.yaml
      dataset_card.json
      validation_report.json
```

### Required files

| File | Description |
|---|---|
| `data/shard-NNNNNN.tar` | Canonical WebDataset shards (all cells, not split-partitioned) |
| `splits/train.json` | Ordered list of global_idx values assigned to the train split |
| `splits/val.json` | Ordered list of global_idx values assigned to the val split |
| `splits/test.json` | Ordered list of global_idx values assigned to the test split |
| `splits/split_membership.parquet` | Per-cell split assignment: global_idx, sample_id, cell_id, split |
| `splits/split_report.json` | Split assignment method, seed, per-sample split counts |
| `gene_panel.json` | Ordered list of gene names matching expression vector indices |
| `gene_panel.sha256` | SHA-256 hash of gene_panel.json for reproducibility |
| `task_config.yaml` | Task type, input/target modalities, gene panel ref, n_genes |
| `loader_config.yaml` | Shard glob pattern(s), split metadata paths, batch size recommendations, num_workers |
| `dataset_card.json` | Dataset provenance: n_cells, n_genes, sample_ids, creation date, daas version |
| `validation_report.json` | Per-split cell counts, gene panel consistency check, shard integrity |

### Required per-cell shard content

Each tar entry must contain:
- `{key}.jpg` — JPEG-encoded patch image
- `{key}.expr.npz` — sparse expression target (`indices`, `values`, `n_genes`)
- `{key}.json` — metadata with required fields:

```json
{
  "global_idx": 12345,
  "sample_id": "A_002",
  "cell_id": "cell_000123",
  "task": "he2st",
  "n_genes": 313,
  "gene_panel_sha256": "abc123..."
}
```

Note: `split` is **not** embedded per-cell in shards. Split assignment is determined at
loader time from `splits/split_membership.parquet` or `splits/{split}.json`.

### Required json fields

| Field | Type | Description |
|---|---|---|
| `global_idx` | int | Row index in the L3 compiled manifest/h5ad |
| `sample_id` | str | Sample identifier |
| `cell_id` | str | Original cell identifier from SpatialData |
| `task` | str | Task type (e.g. `"he2st"`) |
| `n_genes` | int | Length of expression vector |
| `gene_panel_sha256` | str | SHA-256 of `gene_panel.json` for loader-side validation |

## Summary

> Patch extraction (L2) is patch-compiled, not training-ready.
> Compiled h5ad + manifest (L3) is dataset-compiled, not necessarily training-ready.
> Splits are metadata applied by the loader at runtime — not physical shard partitions.
> Training-ready (L4) requires task-specific canonical storage, split metadata, gene panel,
> loader config, and validation artifacts.

See `references/artifact-levels.md` for the full level hierarchy.
See `references/task-adapters.md` for task-specific L4 layouts.
